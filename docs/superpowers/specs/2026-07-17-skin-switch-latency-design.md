# Codex 皮肤切换、玻璃质感与安装可靠性优化设计

## 目标

消除 `v5.0.0` 中健康后台巡检持续扩张耐久锁链、占满单核并阻塞主题切换的问题，修复助手消息区域被渲染为硬边浅色矩形的问题，并补齐 macOS 首次安装、可靠启动、失败诊断和桌宠生效提示，同时保留状态变更、崩溃恢复和跨进程互斥的安全边界。

成功标准：

1. 健康且状态一致的后台巡检不获取耐久操作锁，不创建新的 `operation.lock.*` 文件。
2. 主题、常驻、暂停、恢复等状态变更继续使用现有耐久操作锁。
3. 检测到进程、端口、会话、主题或 renderer 状态漂移时，巡检回退到现有加锁协调流程。
4. 已积累的长锁链在下一次受控状态变更或控制器启动时被压缩，不要求用户手工删除锁文件。
5. 主题切换正常路径不触发前端 `3 秒`超时或 `15 秒`兜底队列。
6. 助手消息的整行布局容器保持透明，不再形成覆盖背景图的硬边浅色矩形。
7. 用户气泡、输入框和审批卡片保留玻璃层，但填充透明度明显低于当前 `88%`，背景图能透过且文字仍清晰。
8. 代码块、工具结果和文件卡片保留独立的可读表面，不因移除助手整行底色而失去层级。
9. 全新 macOS 用户不需要手工创建状态目录或修改权限，安装器能安全创建权限为 `0700` 的可信状态根目录。
10. 已有 Codex 进程退出后，启动器能确认调试参数实际进入新的 Codex 主进程，而不是仅确认 `/usr/bin/open` 返回成功。
11. 生命周期失败同时提供人类可读阶段、可执行下一步和受控技术诊断，不再只显示笼统的 `LIFECYCLE_FAILED`。
12. 桌宠安装结果明确返回是否需要重启，并在命令行入口直接提示用户。

## 方案选择

采用「无锁健康快路径，加锁修复慢路径」。

不采用单纯拉长巡检间隔，因为它只能推迟锁链膨胀。也不在本次重写整个操作锁实现，因为状态迁移和崩溃恢复依赖它，改动面过大。

## 数据流

后台控制器每轮巡检先执行只读快照：

1. 读取 authoritative state 和 session。
2. 识别当前 Codex 进程并验证 CDP 端口归属。
3. 检查现有控制端点仍然可用。
4. 读取 renderer 状态并判断是否完全健康。
5. 再次读取 state、session 和进程身份，确认快照期间没有变化。

若两次快照一致且 renderer 健康，直接返回 `idle`。任何读取失败、身份变化、状态不一致、控制端点缺失或 renderer 漂移，都回退到现有 `withLease("controller:tick")` 协调流程。

主题切换、常驻开关和恢复操作不走快路径，继续使用耐久锁和 revision 校验。

主题切换的视觉提交与耐久提交分离：

1. 用户点击正式主题或原生界面后，当前 renderer 立即应用目标 CSS，但暂不写入本地选择记录，也不向其他窗口广播成功。
2. 后台控制端点继续在耐久锁和 revision 校验内提交 authoritative state。
3. 后台确认后再写入兼容选择记录并广播；拒绝、冲突或超时则恢复点击前的主题。
4. 该顺序只优化感知延迟，不把未确认的视觉状态冒充成已保存状态。

## 玻璃表面分层

当前 CSS 把 `.composer-surface-chrome`、`[data-user-message-bubble]`、`[data-local-conversation-final-assistant]` 和审批表面放在同一组规则中，统一使用 `88%` 的实体色、阴影和 `18px` 模糊。

`[data-local-conversation-final-assistant]` 实际是宽度覆盖整列、高度随整条回复增长的布局容器，不是消息卡片。运行时计算样式显示它的边框宽度为 `0px`、圆角为 `0px`、背景实体色占比为 `88%`。因此用户看到的“锋锐白边”并非真实边框，而是高不透明背景在零圆角容器边缘发生的突变。

修复采用分层规则：

1. 助手整行布局容器显式使用透明背景，不添加边框、阴影或 backdrop filter。
2. 用户消息气泡、输入框和审批表面使用独立的玻璃规则，降低实体色占比，保留柔和模糊、低对比度描边和原生圆角。
3. 代码块、工具结果、文件卡片等内容表面不继承助手整行透明规则，继续使用各自的背景和边界。
4. 不用单纯增加 blur 掩盖问题。高不透明填充会先遮住背景，blur 再大也不会产生真实通透感。

玻璃层的具体透明度以实机可读性为准，初始实现范围为 `52%` 至 `64%`。最终值必须同时通过浅色背景、深色背景和高细节图片背景验收。

## 外部安装反馈判定

外部用户反馈中的问题按证据分为三类：

1. 已由当前代码确认：首次安装先获取 `stateRoot/macos-install-operation` 的子目录锁，基础 `stateRoot` 不存在时会触发 `LOCK_PATH_INVALID`；桌宠安装返回值没有 `restartRequired`，命令行也没有生效提示；生命周期对话框只展示固定安全文案。
2. 有明确风险但需要实机回归：当前 macOS 生命周期通过 `/usr/bin/open -na ... --args` 启动。即使 `open` 成功，Electron 单实例接管仍可能使调试参数没有进入最终主进程。
3. 暂不归因于项目：反馈中的首次 `git clone` 工作树异常。当前本地仓库的 `origin/HEAD` 正确指向 `origin/main`，单份记录不足以区分网络中断、Git 客户端状态或远端瞬时问题。本轮只在安装文档中保留下载失败排查，不修改仓库安装逻辑。

## 首次安装安全引导

在任何嵌套操作锁之前增加专用的状态根目录引导：

1. 逐级验证 `stateRoot` 的父目录是真实目录且不是符号链接。
2. `stateRoot` 不存在时，原子创建并设置权限 `0700`。
3. `stateRoot` 已存在但权限过宽时，只有在它属于当前用户、不是符号链接且祖先可信时才收紧为 `0700`。
4. 所有权不符、路径类型异常或祖先不可信时继续拒绝操作，不用“自动修复”绕过安全边界。
5. 完成引导后再创建 `macos-install-operation` 并进入现有事务锁。

安装器需输出“状态目录已创建”“权限已安全收紧”或具体拒绝原因，但不要求普通用户理解内部锁实现。

## macOS 可靠启动与应用验证

现有“一键应用”流程保留，但启动成功的定义从“启动命令已返回”提升为“精确进程与 renderer 已验证”：

1. 请求当前 Codex 正常退出，确认原进程身份消失且目标端口释放。
2. 从已通过签名校验的 Codex App 解析 `Contents/MacOS/ChatGPT`。
3. 使用 Node `spawn` 和参数数组直接启动该可执行文件，不经过 shell，不依赖 `nohup` 字符串拼接。
4. 验证新主进程的可执行路径、启动时间、调试参数、CDP 端口所有者和进程树归属。
5. 端口就绪后执行主题应用，并验证主 renderer 的 `installed`、`menu`、`themeId`、`revision` 和 `persistenceEnabled`。
6. 任一步失败时沿用现有补偿语义，尽量恢复启动前的关闭或原生运行状态。

不把“直接启动可执行文件”当成放宽校验。签名、路径、进程身份、端口归属和 renderer 状态必须全部成立。

## 分层错误与诊断

生命周期结果增加有限枚举的 `stage` 和结构化 `diagnostics`。至少覆盖：

1. `quit-requested`：是否发送正常退出请求。
2. `old-process-exited`：旧进程是否按精确身份消失。
3. `port-released`：目标端口是否释放。
4. `launch-started`：使用的启动方式和已验证应用路径。
5. `process-verified`：新主进程是否携带目标调试参数。
6. `port-owned`：CDP 端口是否属于该进程树。
7. `renderer-applied`：目标主题是否完成注入并读回一致状态。
8. `compensated`：失败后是否恢复启动前状态。

用户界面采用双层错误：

1. 第一层给出失败阶段、当前状态和一条安全的下一步操作。
2. 第二层在权限为 `0600` 的结果文件或工作室日志中保留错误码、阶段和受控诊断字段。

不把任意底层异常、环境变量或完整命令行原样塞进弹窗，避免诊断改进变成信息泄露。

## 桌宠生效契约

桌宠安装继续只负责校验资源、事务写入和选择配置，不在本轮静默重启 Codex。返回结果增加：

1. `restartRequired`：配置发生变化且运行中的 Codex 无法热刷新时为 `true`。
2. `effectivePetId`：最终写入并验证的桌宠 ID。
3. `nextAction`：需要重启时给出明确中文提示。

命令行包装器必须展示该提示。自动重启涉及用户正在进行的对话和任务，不能在没有明确交互确认时执行。

## 后续诊断入口

现有 `doctor` 保留只读运行时诊断。本轮先通过生命周期结果文件补充稳定阶段和受控诊断字段，不扩大命令的输出协议。

统一详细诊断和 `diagnose --fix` 作为下一阶段独立设计，纳入状态目录权限、锁链深度、启动方式、进程调试参数、端口归属、renderer 计算状态和桌宠配置。修复模式会同时修改权限、锁状态和进程生命周期，必须先定义逐项授权、回滚和幂等边界。

## 锁链收敛

生产状态锁使用较低但仍有余量的压缩阈值。控制器启动或首次真实状态变更获取锁时，如果发现旧链已经超过阈值，沿用现有经过测试的压缩协议收敛链条。禁止在相关进程运行时直接删除锁文件。

## 错误处理

快路径必须 fail closed。任何不确定状态都不能宣称健康，也不能修改状态，而是回退到原有加锁流程。若加锁流程仍失败，继续使用现有结构化日志和错误结果。

CDP 端口身份不匹配作为独立故障保留，不通过放宽进程身份校验来换取速度。

## 测试

先新增失败回归测试，证明：

1. 连续健康巡检不会调用 `withLease`。
2. renderer 漂移、状态变化、会话变化、控制端点缺失时仍会调用 `withLease`。
3. renderer 控制请求优先处理，且主题状态变更仍在锁内完成。
4. 生产锁配置能在下一次受控获取时压缩已有长链。
5. 主题点击在后台响应未完成时已更新当前 renderer，但不提前持久化或广播；失败和冲突会恢复原主题。
6. 生成的 CSS 不再把 `[data-local-conversation-final-assistant]` 与玻璃卡片共用背景规则，并对该布局容器显式清除背景和阴影。
7. 用户气泡、输入框和审批表面的实体色占比处于设计范围，且继续声明 blur 和低对比度边界。
8. 实机 DOM 计算样式中，助手整行容器背景透明、无阴影；用户气泡和输入框背景 alpha 不高于 `0.64`。
9. 全新 HOME 下不预建状态目录也能完成安装，最终目录为当前用户所有、权限为 `0700`。
10. 符号链接、非当前用户目录和不可信祖先仍被拒绝；当前用户拥有的真实目录权限过宽时可安全收紧。
11. 模拟 `open` 成功但参数未进入主进程的场景会失败，直接可执行文件启动且进程、端口验证一致的场景通过。
12. 每个生命周期失败阶段都生成稳定错误码、`stage`、补偿状态和一条用户可执行的下一步。
13. 桌宠首次选择和变更返回 `restartRequired: true`，重复安装且配置未变化时返回 `false`。

完成单元测试后运行完整 Node 测试集，再安装到本机稳定目录。实机验收使用全新临时 HOME 覆盖首次安装，使用真实 Codex 覆盖退出、直接启动、端口归属和 renderer 应用，并记录切换耗时、控制器 CPU、锁文件数量、主题状态一致性和三类背景下的玻璃层计算样式。截图仅用于视觉复核，DOM 计算样式作为可重复的硬验收依据。
