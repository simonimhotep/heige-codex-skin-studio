# Codex 皮肤切换延迟优化设计

## 目标

消除 `v5.0.0` 中健康后台巡检持续扩张耐久锁链、占满单核并阻塞主题切换的问题，同时保留状态变更、崩溃恢复和跨进程互斥的安全边界。

成功标准：

1. 健康且状态一致的后台巡检不获取耐久操作锁，不创建新的 `operation.lock.*` 文件。
2. 主题、常驻、暂停、恢复等状态变更继续使用现有耐久操作锁。
3. 检测到进程、端口、会话、主题或 renderer 状态漂移时，巡检回退到现有加锁协调流程。
4. 已积累的长锁链在下一次受控状态变更或控制器启动时被压缩，不要求用户手工删除锁文件。
5. 主题切换正常路径不触发前端 `3 秒`超时或 `15 秒`兜底队列。

## 方案选择

采用「无锁健康快路径，加锁修复慢路径」。

不采用单纯拉长巡检间隔，因为它只能推迟锁链膨胀。也不在本次重写整个操作锁实现，因为状态迁移和崩溃恢复依赖它，改动面过大。

## 数据流

后台控制器每轮巡检先执行只读快照：

1. 读取 authoritative state 和 session。
2. 识别当前 Codex 进程并验证 CDP 端口归属。
3. 检查现有控制端点仍然可用。
4. 读取 renderer 状态并判断是否完全健康。
5. 再次读取 state、session 和进程身份，确认快照期间没有变化。

若两次快照一致且 renderer 健康，直接返回 `idle`。任何读取失败、身份变化、状态不一致、控制端点缺失或 renderer 漂移，都回退到现有 `withLease("controller:tick")` 协调流程。

主题切换、常驻开关和恢复操作不走快路径，继续使用耐久锁和 revision 校验。

## 锁链收敛

生产状态锁使用较低但仍有余量的压缩阈值。控制器启动或首次真实状态变更获取锁时，如果发现旧链已经超过阈值，沿用现有经过测试的压缩协议收敛链条。禁止在相关进程运行时直接删除锁文件。

## 错误处理

快路径必须 fail closed。任何不确定状态都不能宣称健康，也不能修改状态，而是回退到原有加锁流程。若加锁流程仍失败，继续使用现有结构化日志和错误结果。

CDP 端口身份不匹配作为独立故障保留，不通过放宽进程身份校验来换取速度。

## 测试

先新增失败回归测试，证明：

1. 连续健康巡检不会调用 `withLease`。
2. renderer 漂移、状态变化、会话变化、控制端点缺失时仍会调用 `withLease`。
3. renderer 控制请求优先处理，且主题状态变更仍在锁内完成。
4. 生产锁配置能在下一次受控获取时压缩已有长链。

完成单元测试后运行完整 Node 测试集，再安装到本机稳定目录。实机验收同时记录切换耗时、控制器 CPU、锁文件数量和主题状态一致性。
