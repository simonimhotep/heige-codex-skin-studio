# 预设主题固定签名卡模块设计

## 目标

为除原生界面外的所有内置预设主题提供与 Miku 主题现有右下角拍立得一致的签名卡效果。卡片不按主题重新设计完整成品，而是拆成可复用的固定相框、可替换的主题画芯和程序渲染的签名文字。

成功标准：

1. 所有内置预设主题切换后都显示右下角签名卡。
2. 视觉语言以现有 Miku 拍立得为基准，包括竖向比例、轻微倾斜、白色相纸、装饰贴纸和阴影。
3. 卡片主题名随当前主题变化，署名固定且准确显示为 `By@HeiGe`。
4. 默认复用主题 hero 图片作为画芯，不为现有主题重新生成角色图片。
5. 新增主题只提供 hero 即可自动获得签名卡；需要不同构图时可提供可选画芯资源。
6. 卡片不得挡住输入框和主要操作，在窄窗口下自动缩小或隐藏。
7. 主题即时切换性能不得因签名卡退化，画面更新仍应在一次同步渲染内完成。

## 方案选择

### 采用方案：运行时分层签名卡

卡片由两个固定视觉层和一个文本层组成：

1. 画芯层使用当前主题 hero，或使用可选的独立画芯资源。
2. 相框层使用一份共享透明素材，保留 Miku 卡片的相纸、回形针、爱心和星星装饰。
3. 文字层由生成 CSS 写入当前主题名称和固定署名 `By@HeiGe`。

共享相框只打包一次。每个主题不保存重复相框，也不保存一张已经合成好的完整卡片。

### 未采用方案：每个主题预合成完整图片

该方案与现有 `polaroid.webp` 兼容，但会重复保存相同相框。后续增加主题时仍需要重新制作完整图片，不能形成固定模块。

### 未采用方案：每个主题重新生成完整卡片

该方案容易造成相框比例、装饰位置和署名文字漂移，也会增加资产体积和人工验收成本。

## 资源结构

新增一份共享透明相框：

```text
assets/signature-card-frame.png
```

要求：

1. 画布比例为 2:3，与现有 Miku 卡片一致。
2. 中部画芯窗口透明。
3. 下部签名区保留足够留白，但不把主题名或署名烘焙进图片。
4. 素材只包含固定装饰和相框，不包含任何主题人物。
5. 使用 PNG 保留透明通道，文件大小纳入现有资源预算和资产溯源检查。

相框以用户现有 Miku 拍立得为视觉参考制作。可以使用图片编辑能力清除人物画芯和文字，但最终相框必须人工检查透明窗口、边缘完整性和装饰位置。`ASSET_PROVENANCE.md` 必须记录参考素材、派生方式和发布处置，不能因为使用图片编辑模型而跳过资产来源审计。

主题 manifest 增加一个可选字段：

```json
{
  "cardArtwork": "card.webp"
}
```

规则：

1. `cardArtwork` 缺省时复用 hero。
2. 提供时必须通过与 hero 相同的 MIME、尺寸、像素、路径和资源预算验证。
3. 该字段只替换画芯，不允许覆盖相框和署名。
4. 当前 9 个非 Miku 预设不新增 `cardArtwork`，直接使用现有 hero。

## 兼容策略

现有 Miku 主题的 `polaroid.webp` 是完整合成图，继续作为兼容优先级最高的精确视觉基准。

渲染优先级：

1. 主题存在旧版 `polaroid` 时，继续渲染该完整图片。
2. 否则使用共享相框模块。
3. 模块画芯优先使用 `cardArtwork`，没有时使用 hero。
4. 原生界面和用户临时上传主题不自动显示预设签名卡。

这样不会改变 Miku 已被用户确认的效果，也不会破坏第三方主题已经提供的完整拍立得资源。

## 渲染结构

`buildSkinCss` 接收以下数据：

```text
heroDataUrl
cardArtworkDataUrl
polaroidDataUrl
theme.name
```

渲染规则：

1. 旧版完整 `polaroidDataUrl` 存在时沿用当前 `body::after` 单图逻辑。
2. hero data URL 在当前主题 CSS 中只声明一次为 `--heige-hero-image`，主背景和默认画芯共同引用该变量。
3. 模块模式使用 `body::before` 渲染带裁切的画芯和卡片底色。
4. `body::after` 渲染透明相框、主题名称与 `By@HeiGe`。
5. 两层使用相同的固定定位、尺寸、旋转角和 `pointer-events: none`。
6. 图层位于背景之上、主题中心弹窗之下，不拦截点击。

共享相框 data URL 不写入每个主题 CSS。注入器只读取和编码一次相框，菜单运行时创建独立共享样式节点，在根元素声明 `--heige-signature-card-frame-image`。所有主题通过该 CSS 变量引用同一份相框。重注入和卸载必须清理旧共享样式，防止资源残留。

画芯使用 `object-fit: cover` 等价的 CSS 背景裁切。默认焦点为中心偏上，以减少人物面部被裁掉的概率。后续如有明确需求，再增加焦点配置，本次不扩展 manifest。

## 响应式与可读性

桌面大窗口：

1. 卡片宽度约 190 至 210 px，高度按 2:3 比例。
2. 右边距 18 至 24 px，底边距 22 至 30 px。
3. 使用轻微逆时针旋转和主题色阴影。

中等窗口：

1. 卡片按视口宽度缩小至约 150 px。
2. 保持右下角，不覆盖输入框主要操作区域。

窄窗口：

1. 当视口宽度不足 900 px 或高度不足 650 px 时隐藏签名卡。
2. 不允许通过压缩到不可读尺寸继续显示。

文字：

1. 主题名称最多显示一行，超出时截断。
2. 署名固定为 `By@HeiGe`，不经过生图模型。
3. 颜色根据主题强调色和相纸底色计算，保证可读性。

## 数据流

1. `theme-schema` 验证 hero、旧版 polaroid 和可选 cardArtwork。
2. `injector` 读取当前主题资源，并读取一份共享相框资源。
3. 所有资源先完成大小和图片元数据验证，再编码为 data URL。
4. `skin-menu` 把共享相框写入一个独立样式节点，整个菜单 payload 只保留一份相框 data URL。
5. `skin-css` 根据兼容优先级生成旧版单图或新版分层 CSS，hero data URL 在单个主题 CSS 内只出现一次。
6. 乐观主题切换继续同步替换整段主题 CSS，签名卡与背景在同一帧更新。
7. 后台保存确认不参与签名卡绘制，因此不会增加切换等待时间。

## 错误处理

1. 共享相框缺失、为空、越界或验证失败时，主题应用失败，不显示一个没有边框的残缺画芯。
2. 可选 cardArtwork 无效时，整个主题验证失败，不静默使用可能已漂移的资源。
3. hero 有效但默认裁切不理想属于视觉问题，不属于运行错误，可通过后续增加 cardArtwork 修正。
4. 旧版 polaroid 保持现有严格验证和失败边界。

## 测试

自动化测试至少覆盖：

1. manifest 接受合法 cardArtwork，并拒绝非法路径、格式、尺寸和预算超限。
2. 共享相框被严格读取、验证和纳入注入资源预算。
3. Miku 继续走旧版完整 polaroid 路径。
4. 其余预设走共享相框和 hero 画芯路径。
5. 提供 cardArtwork 时优先于 hero。
6. 生成 CSS 包含主题名称和准确的 `By@HeiGe`，不包含外部 URL。
7. 卡片两层点击穿透，z-index 低于主题中心。
8. 窄窗口隐藏规则生效。
9. 连续主题切换后不存在旧画芯或旧相框残留。
10. 全量测试、确定性安装包测试和本机真实切换验收继续通过。
11. 共享相框的透明窗口包含真实 alpha 像素，且文件格式、尺寸和资源哈希稳定。
12. 菜单注入表达式中共享相框 data URL 只出现一次，每个主题的 hero data URL 在对应 CSS 中只出现一次。

## 真实验收

本机安装后至少完成：

1. 依次切换 Miku、一个浅色预设和一个深色预设。
2. 确认 Miku 外观不变。
3. 确认浅色和深色预设都显示固定相框、正确画芯和 `By@HeiGe`。
4. 确认卡片不遮挡输入框、主题入口和主题中心弹窗。
5. 确认主题切换即时渲染耗时仍低于 20 ms。
6. 最后恢复到用户当前主题和原有常驻状态。
